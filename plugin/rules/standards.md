# thinking-tree 编排标准 v0.2

边用边迭代。

## 四个空间

```
~/.thinking-tree/
├── 思路体系（*.md）── 结晶态，有主线（理解）
├── fragments.md    ── 有价值但还没归位（观察）
├── questions.md    ── 明确的"不知道"（探索）
└── todos.md        ── 想清楚后的行动意图（行动）
```

## 路由规则

| 如果… | 则放入… |
|-------|---------|
| 一个独立的点，脱离上下文能看懂 | 碎片池 |
| 有主线，能串起多个点成一篇 | 思路文件 |
| 明确知道自己不知道什么，有方向性 | 开放问题 |
| 从思考自然衍生的"接下来要做"，具体且可执行 | 行动项 |
| 项目工程相关，不是产品思考 | 项目 .claude/ 体系（不进 thinking-tree） |

### 边界冲突

- 碎片 vs 思路 → 先放碎片，积累够了再提炼
- 碎片 vs 问题 → 有明确的"不知道"就是问题，否则是碎片
- 已有思路文件能接上 → 进该思路文件，不新建碎片

### 示例

**碎片** ✓：「AI 是宝石，但宝石不能直接用——不同场景需要不同的适配装备」
— 独立的一个点，不需要上下文。

**思路文件** ✓：「理解 = 信号 × 语境 → 有损转换 → 光谱模型 → 语境结构 → 产品启示」
— 有主线，多个点串成一篇（information-structure.md）。

**开放问题** ✓：「语义信息能量化吗？语法层可以用 Shannon 熵，语义层呢？」
— 明确的"不知道"，有具体方向。

**行动项** ✓：「验证自动策展的性能提升：模糊想法→拆解→自动组织的总耗时 vs 手动整理」
— 从思路《自动策展的价值定位》衍生，具体可执行，有明确的验证方向。

**不进 thinking-tree** ✗：「前端右侧从 card+tree 改为三空间视图」
— 项目工程决策，进 DECISIONS.md。

## 流转

```
碎片积累 ──→ 提炼成思路文件
思路中发现未解 ──→ 进 questions.md
问题解答后 ──→ 结论进思路体系，问题标记 ✓
思路/碎片/问题 ──→ 衍生行动项 ──→ 执行后反馈进各空间
```

## 各空间质量标准

### 行动项
- 不是"待做事项"，而是**思考的产出形态**
- 具体到能执行（包含条件/目标/验证方式）
- 来源明确：来自某个具体的思路/碎片/问题
- 非工程类任务（工程任务进 DECISIONS.md）

### 思路文件
- 主线清晰：从哪来 → 到哪去，每节在主线上有位置
- 不留散尾巴：未解问题去 questions.md，未归位的想法去 fragments.md
- 读完能一句话概括：概括不了 = 主线不清晰或塞了太多主题
- 视觉优先：ASCII 图表达关系，短句表达结论，段落只在必要时用

### 碎片
- 一个标题 + 一段话 + 可选的示意图
- 带 #标签 方便归类
- 不要求完整，但要求有明确的一个点

### 开放问题
- 一个具体的疑问（不是模糊的"待研究"）
- 附上来源上下文（从哪个讨论中产生的）
- 解答后结论进思路体系，问题标记 ✓
