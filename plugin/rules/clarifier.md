# 思考记录

> 本规则由 /think 命令切换启用/禁用。

## 每轮强制检查（必须执行）

每轮回复结束前，你**必须**判断："本轮对话是否产生了值得记录的独立认知？"

**审视范围包括：**
1. 用户消息中显式提出的观点、判断、方向
2. **你自己的回复中产生的独立认知**——分析结论、架构判断、新发现的关联。AI 回复中的精华往往比用户的提问更值得捕获
3. 对话碰撞中浮现的产品认知、方法论观点，即使主题是工程操作

**原则：宁可多记再删，不可漏记。** 未记录的认知是不可挽回的损失，多记了的碎片可以在 /reduce 时清理。

判断结果：
- 有独立的点 → 写碎片（fragments.md）
- 有明确的"不知道"，有具体方向 → 写问题（questions.md）
- 都没有 → 不记录

## 路由

| 性质 | 去向 | 示例 |
|------|------|------|
| 独立观点，脱离上下文能看懂 | fragments.md | 「AI 是宝石，但不同场景需要不同的适配装备」 |
| 明确的疑问，有具体方向 | questions.md | 「语义信息能量化吗？语法层可以用 Shannon 熵，语义层呢？」 |

### 边界判断
- 碎片 vs 问题 → 有明确的"不知道"就是问题，否则是碎片
- 已有思路文件能接上 → 进该思路文件，不新建碎片
- 不确定 → 先放碎片，积累够了再提炼

## 记录操作

通过原子写入脚本 `write-item.js` 执行，**不要直接 Edit/Write 文件**。

脚本位于插件 `scripts/write-item.js`。用 Bash 调用：

```bash
echo '{"type":"fragment","title":"标题（日期）","body":"一段话描述。"}' | node ~/.thinking-tree/bin/write-item.js
```

- `type`：`"fragment"` | `"question"` | `"todo"`
- `title`：标题文本（不带 `## ` 前缀）
- `body`：内容文本
- 碎片的 frag ID 和 meta.json 由脚本自动管理

问题的 body 格式：`来源：xxx\n具体描述。`

## 反馈

仅在**回复用户消息**时，末尾附反馈块。格式：

记录了内容时：
```
---
📝 记录碎片：「标题」
   → 核心观点一句话概述
📝 记录碎片：「标题2」
   → 核心观点一句话概述
```

没有记录时：
```
---
🌳 本轮无新认知
```

**要求：**
- `→` 后面必须是这条碎片的核心观点概述，让用户一眼知道记了什么
- 有多条就列多条
- 反馈块用 `---` 与正文隔开

**不附反馈的情况**：Hook 输出（tsc 结果、session-log、插件日志）、系统通知、tool 结果。这些不是用户消息，**不回应、不附反馈**。

## 碎片质量标准

记录的碎片应该是**思考结晶**，不是对话摘要。

**好碎片的特征：**
- 一个独立的洞察/判断，脱离上下文仍有价值
- 有结构：观点 + 原因/推导 + 启示/推论
- 适当使用 ASCII 图、对比表、层级关系来表达结构
- 标题是锐利的一句话判断，不是"关于 XXX 的讨论"

**坏碎片的特征（不要记这种）：**
- "我们讨论了 XXX" — 这是日志不是碎片
- 只有一句话没有展开 — 要么展开，要么不记
- 重复已有碎片的内容
- 临时想法、不成型的片段
- 工程操作本身（但工程中浮现的认知**必须**捞出来）

## 纠正协议

| 用户说 | 做 |
|--------|-----|
| "不对" | 问哪里不对 |
| "移到碎片/问题" | 直接移 |
| "这个不用记" | 删除 |
| "漏了 XXX" | 补记 |
