# 思考记录

> 本规则由 /think 命令切换启用/禁用。
> 状态由 UserPromptSubmit hook 每轮注入：**只有看到 `/think: ON` 时才执行本文件规则**，其他情况（OFF 或无状态）一律跳过，不记录、不附反馈。

## 上下文检索

回应用户前，判断当前话题是否可能与 ~/.thinking-tree/ 下已有思考相关。如果相关，先用 Glob 查看 `~/.thinking-tree/*.md` 思路文件列表，再用 Read 读取相关文件，基于已有认知回应，避免重复或遗漏已有判断。

## 每轮强制检查（必须执行）

每轮回复结束前，你**必须**审视整个对话至此，判断："是否有尚未记录的认知结晶？"

不只看当前这一轮——有些认知是多轮碰撞逐步浮现的，没有单一轮次是"产生"的那一刻。

**审视范围包括：**
1. 用户消息中显式提出的观点、判断、方向
2. **你自己的回复中产生的独立认知**——分析结论、架构判断、新发现的关联
3. **多轮讨论累积浮现的认知**——前几轮分别讨论了不同侧面，到此刻拼成了完整洞察

**原则：宁可多记再删，不可漏记。** 未记录的认知是不可挽回的损失，多记了的碎片可以在 /reduce 时清理。

判断结果：
- 有独立的点 → 写碎片（fragments.md）
- 有明确的"不知道"，有具体方向 → 写问题（questions.md）
- 都没有 → 不记录

## 用户偏好

如果 `~/.thinking-tree/.preferences.md` 存在，每轮判断时**必须用 Read 工具从磁盘重新读取**（不能依赖 context 里的缓存副本——偏好可能刚被 /pref 更新，session 内任何时刻都可能变）。

偏好作为**额外判断依据**叠加在以上通用规则之上。用户偏好优先于通用规则（如用户说"不记 debug"，即使 debug 中有认知也不记）。通过 `/pref` 命令管理。

## 路由

| 性质 | 去向 | 示例 |
|------|------|------|
| 独立观点，脱离上下文能看懂 | fragments.md | 「AI 是宝石，但不同场景需要不同的适配装备」 |
| 明确的疑问，有具体方向 | questions.md | 「语义信息能量化吗？语法层可以用 Shannon 熵，语义层呢？」 |

### 边界判断
- 碎片 vs 问题 → 有明确的"不知道"就是问题，否则是碎片
- 已有思路文件能接上 → 进该思路文件，不新建碎片
- 不确定 → 先放碎片，积累够了再提炼

## 记录操作

通过原子写入脚本 `write-item.js` 执行，**不要直接 Edit/Write 文件**。

脚本位于插件 `scripts/write-item.js`。用 Bash 调用：

```bash
echo '{"type":"fragment","title":"标题（日期）","body":"一段话描述。"}' | node ~/.thinking-tree/bin/write-item.js
```

- `type`：`"fragment"` | `"question"` | `"todo"`
- `title`：标题文本（不带 `## ` 前缀）
- `body`：内容文本
- 碎片的 frag ID 由脚本自动管理

问题的 body 格式：`来源：xxx\n具体描述。`

## 反馈

仅在**回复用户消息**时，末尾附反馈块。格式：

记录了内容时：
```
---
📝 记录碎片：「标题」
   → 核心观点一句话概述
📝 记录碎片：「标题2」
   → 核心观点一句话概述
```

没有记录时：
```
---
🌳 本轮无新认知
```

**要求：**
- `→` 后面必须是这条碎片的核心观点概述，让用户一眼知道记了什么
- 有多条就列多条
- 反馈块用 `---` 与正文隔开

**不附反馈的情况**：Hook 输出（tsc 结果、插件日志）、系统通知、tool 结果。这些不是用户消息，**不回应、不附反馈**。

## 碎片质量标准

记录的应该是**思考结晶**（可迁移的认知），不是**操作日志**（具体的经验）。

### 核心判断：可迁移性测试

写碎片前问自己：**删掉所有具体任务/项目/库/API 的名字，这条内容还有意义吗？**

- 有意义 → 认知结晶，记录
- 没意义（脱了具体上下文就是空话）→ 操作日志，不记录

### 正反对比

| 记 ✅ 可迁移认知 | 不记 ❌ 操作日志 |
|-----------------|-----------------|
| 手搓验证的"够了"信号 = 覆盖所有核心技术点后边际递减 | Fish Audio + Node fetch 的三个坑 |
| 视频 pipeline 的音画对齐是两层问题 | Remotion useVideoConfig 的三个用法 |
| 讲解视频的正确架构：章节大纲统领 | 我们今天调通了 Whisper 的时间戳 |
| Remotion 的魔法：把视频翻译成前端开发 | 搭建 Node fetch 踩的坑列表 |

**左栏共同特征**：删掉具体名词后仍是完整判断（方法论/架构洞察/品类认知）
**右栏共同特征**：删掉具体名词后变成空话（只对做过这个任务的人有意义）

### 好碎片的特征

- 一个独立的洞察/判断，**半年后脱离具体任务语境仍有价值**
- 图示优先：优先用 ASCII 示意图、对比表、层级关系来表达核心关系和逻辑，文字只用于图示无法表达的部分
- 标题是锐利的一句话判断，不是"关于 XXX 的讨论"

### 坏碎片的特征（不要记这种）

- "我们讨论了 XXX" — 这是日志不是碎片
- **具体操作记录/bug 清单** — "X 的三个坑"、"遇到的问题列表"，是操作日志不是认知
- **任务特定的经验** — 删掉具体库/API/项目名就没意义（"Fish Audio 的坑"、"Remotion 的三个用法"）
- 只有一句话没有展开 — 要么展开，要么不记
- 重复已有碎片的内容
- 临时想法、不成型的片段
- 工程操作本身（但工程中浮现的**可迁移认知**必须捞出来）

## 纠正协议

| 用户说 | 做 |
|--------|-----|
| "不对" | 问哪里不对 |
| "移到碎片/问题" | 直接移 |
| "这个不用记" | 删除 |
| "漏了 XXX" | 补记 |
